# ADR-Driven Development Workflow

본 저장소의 모든 의미 있는 설계 결정은 결과만 코드/문서로 남기는 것이 아니라, "왜 이 결정이 내려졌는가"의 흔적까지 베스트 프랙티스 시연의 일부로 간주하여 ADR(Architecture Decision Record)로 박힌다.

> **이 스킬은 프로젝트 내에서 새로운 기능 설계, 아키텍처 변경, 주요 기술 스택 도입 등 '결정'이 필요한 모든 작업에 강제 적용된다.**

## 1. 진행 방식 (Task Sequence)

모든 작업은 코드를 즉시 작성하기 전에 아래의 순서를 반드시 준수해야 한다. AI가 단독으로 결정하고 진행해서는 안 된다.

| 단계 | 역할 (누가) | 설명 |
|---|---|---|
| 1. 결정 지점 식별·옵션 정리·트레이드오프 제시 | **AI** | 문제를 분석하고 가능한 해결 옵션들과 각각의 장단점(Trade-off)을 사용자에게 제안한다. |
| 2. 결정 | **사용자** | AI가 제시한 옵션을 바탕으로 사용자가 최종 선택을 내린다. |
| 3. 결정 근거 검증 | **함께** | 사용자의 결정을 환원점(단순성/목표 부합 여부) 및 3 침범 카테고리(아키텍처/의존성/컨벤션 훼손 여부)로 함께 검증한다. |
| 4. 산출물(ADR) 작성 | **AI** | 검증이 완료되면, 사용자의 결정을 반영하여 아래 형식에 맞춰 ADR 문서를 작성한다. |

## 2. ADR 문서 형식

ADR 산출물은 `agent-log/ADR/` 폴더 하위에 생성되며, 반드시 다음 구조를 따라야 한다.

- **맥락 (Context)** — 어떤 문제·압력이 있어 이 결정이 필요했는가
- **결정 (Decision)** — 무엇을 결정했는가
- **근거 (Rationale)** — 왜 이 결정인가 (트레이드오프 비교 내용 포함)
- **결과 (Consequences)** — 긍정·부정 영향, 후속 결정에 미치는 제약
- **관련 (Related)** — 인접 ADR (관련된 기존 결정이나 연관 이슈)

---
**주의사항**: 코드를 작성하기 전에, 먼저 위 프로세스를 통해 구조와 방향성에 대해 사용자와 합의를 마쳐야 한다.
